CONCEPT SUMMARY
--------------------------------------------------------------------------------
A mobile IDE (Android) that runs a multi-agent micro-swarm of small, specialized
AI models entirely on-device. The app writes, executes, validates, and repairs
code autonomously — all offline, on an Arm-powered phone.

The "WOW": An Android phone in airplane mode writes code, catches its own errors,
repairs them, and runs the final result — no internet, no cloud, no laptop.
Works even on a budget 2019 phone with 4GB RAM.

The "OPTIMIZATION": Multiple ultra-compact models (50M-500M params) quantized to
INT4/INT8, exported via ExecuTorch with KleidiAI acceleration, running sequentially
with flat memory ceiling on mid-range Arm devices. Battery-efficient inference
using adaptive core selection and thermal-aware scheduling.

ARCHITECTURE OVERVIEW (7 LAYERS)
--------------------------------------------------------------------------------

LAYER 1: USER INTERFACE (The Presentation)
- Kotlin + Jetpack Compose + Material 3
- Dark theme, code-focused, Cursor/Windsurf-inspired layout
- Chat panel (left) + code editor panel (right) = split view
- Code editor with syntax highlighting (tree-sitter via ComposeSyntaxHighlighter)
- Console output + error display (bottom panel)
- Agent status indicators: each agent shown in different color
  - Architect = blue, Coder = green, Validator = orange
  - Active agent highlighted, memory usage displayed
  - Agent name hover shows current task
- Settings (model selection, quantization level, power saving mode)
- Battery level indicator + energy consumption tracking

LAYER 2: THE BRAIN / ORCHESTRATION LAYER (Swarm Orchestrator)
- Deterministic state machine (NOT AI-driven — rigid scheduling)
- Sequential Execution Loop: Load Agent → Execute → Unload → Next Agent
- Active Memory Pruning: purge model weights after each agent runs
- Flat memory ceiling regardless of swarm complexity
- Manages: Architect → Coder → Validator → (Repair loop)
- Thermal monitor: reduces threads/clock when throttling detected
- Adaptive core selection: big cores for prefill, little cores for decode

LAYER 3: CORE AI LAYER (The Micro-Swarm)
- Agent 1 (Architect): Breaks down user prompt into structured plan
- Agent 2 (Coder): Generates code from structured plan
- Agent 3 (Validator): Analyzes errors, generates repair instructions
- Each agent: 50M-500M param model, task-specific fine-tuned
- INT4/INT8 quantized, exported to .pte via ExecuTorch
- Supports both AI-generated code and user manual editing

LAYER 4: CORE INFERENCE & OPTIMIZATION LAYER
- ExecuTorch runtime (50KB footprint) [PENDING USER CONFIRMATION]
- KleidiAI micro-kernels (NEON/i8mm/SME2 acceleration)
- XNNPACK backend for Android CPU
- Ahead-of-time compiled .pte model graphs
- mmap loading for memory efficiency (LOAD_MODE_MMAP default)
- Runtime feature detection: NEON (always), i8mm (if available), SME2 (if available)
- Graceful fallback: SME2 → i8mm → NEON baseline

LAYER 5: INTEGRATION BRIDGE
- JNI bridge between Kotlin app and C++ ExecuTorch runtime
- ExecuTorch AAR library: org.pytorch:executorch-android (Maven Central)
- Model loading/unloading lifecycle management
- Token streaming from inference to UI
- Error trace capture from sandbox to validator agent
- C++ native code for hardware-level optimization (NDK)

LAYER 6: APPLICATION SANDBOX ENVIRONMENT (The Runtime)
- Local code execution environment (isolated)
- Language support at launch: Python (Chaquopy) + JavaScript (QuickJS)
- Future languages: Java, C, HTML/CSS (if time permits)
- Captures stdout, stderr, exit codes
- Timeout enforcement + crash isolation
- No network access (true offline validation)
- Feeds errors back to Validator agent for repair loop
- tree-sitter for syntax validation before execution
- HARDWARE BRIDGE (Phase 2 — see HARDWARE_BRIDGE.md):
  AI-generated code can call Android hardware APIs (camera, flashlight,
  GPS, sensors, audio, storage) and the on-device LLM itself (ai.ask()).
  Bridge objects injected into sandbox runtime as global variables.
  Self-repair loop catches hardware API errors (permission denied,
  sensor unavailable, invalid params) and feeds them back to LLM.
  Language-agnostic: same bridge API for JS, Python, future languages.
  NOT the core submission — built on top of working self-repair loop
  as a WOW-factor multiplier. Build only after Phase 1 is complete.

LAYER 7: BENCHMARKING & PROOF ARTIFACTS
- Built-in benchmark mode: tokens/sec, TTFT, memory RSS, model size, energy/token
- Before/after optimization comparison logs
- Auto-generate proof artifact screenshots for submission
- Battery consumption tracking (Android BatteryManager API)
- Thermal throttling curve (performance over time)
- Compare: FP32 vs INT8 vs INT4, with/without KleidiAI, big vs little cores
- Live demo mode: airplane mode → generate → execute → repair → run

--------------------------------------------------------------------------------
THE AUTONOMOUS VALIDATION LOOP (CLOSED-LOOP RESILIENCY)
--------------------------------------------------------------------------------

[User Prompt] → [Architect Agent: Plan] → [Coder Agent: Generate Code]
       ↑                                              ↓
       |                                    [Sandbox: Execute Code]
       |                                              ↓
[Coder Agent: Repair] ← [Validator Agent: Analyze] ← [Pass?]
                                                          |
                                                     [YES → Output]

Loop continues autonomously until:
  1. Code executes successfully → display result
  2. Max iterations reached → show best attempt
  3. No improvement detected → show error + last attempt

Supports: simple scripts, mid-complexity code, multi-file projects (sequential)

MODEL STRATEGY
--------------------------------------------------------------------------------

PRIMARY CANDIDATE: Qwen2.5-Coder-0.5B-Instruct [PENDING USER CONFIRMATION]
  - 0.49B params, Apache 2.0 license
  - HumanEval: 28% pass@1, MBPP: 52.9% pass@1
  - 32K context length
  - Already tested on Android via ExecuTorch and LiteRT
  - Samsung S24 Ultra: 251 prefill tk/s, 30 decode tk/s (CPU, LiteRT)
  - INT4 quantized: ~130-280MB

ALTERNATIVE FOR LOW-END DEVICE (LG Q70, 4GB RAM): SmolLM2-135M-Instruct
  - 135M params, ultra-compact
  - INT4 quantized: ~35-70MB → fits in L3 cache
  - Safer for 4GB RAM devices
  - Less code-capable than Qwen2.5-Coder

FALLBACK: Llama 3.2 1B Instruct
  - Official ExecuTorch support, pre-built .pte files on Hugging Face
  - Larger but well-tested export pipeline
  - Arm's own learning paths use this model

TRAINING/FINE-TUNING PLAN:
  - Start with pre-trained model
  - LoRA fine-tune for each agent role:
    - Architect: fine-tune on planning/task decomposition data
    - Coder: fine-tune on code generation from structured specs
    - Validator: fine-tune on error analysis + repair instructions
  - Use agent distillation: generate training data from large model
    (e.g., Qwen2.5-Coder-32B as teacher → 0.5B as student)
  - Post-training quantization: INT4 groupwise (group_size=32/64)
  - Export to .pte via ExecuTorch with XNNPACK + KleidiAI
  - Training hardware: A100/L40 cloud GPU or 4080M local (all sufficient)

MODEL DEPLOYMENT OPTIONS:
  Option A: One base model + 3 LoRA adapters (swapped at runtime)
    - Base INT4: ~130-280MB + 3 LoRA adapters: ~15-45MB = ~300MB total
    - KEY optimization story: 70% memory reduction vs 3 separate models
    - Requires ExecuTorch runtime LoRA support (NEEDS VERIFICATION)
  Option B: Three separate .pte files, loaded sequentially
    - 3 × 130-280MB on disk, only 1 in memory at a time
    - Simpler, guaranteed to work
    - Memory ceiling = size of largest single model

--------------------------------------------------------------------------------
BATTERY EFFICIENCY STRATEGY (TRACK 3 REQUIREMENT)
--------------------------------------------------------------------------------

1. Sequential execution = lower peak power (only 1 model active)
2. INT4 quantization = 8x less data movement = proportional energy savings
3. Adaptive core selection: big cores for prefill, little cores for decode
   - Decode is memory-bound, doesn't need big cores
   - 23% energy savings with no speed loss (MNN-AECS research)
4. Thermal-aware inference: monitor temp, reduce threads when throttling
5. mmap loading: only load needed weights into memory
6. Small models in cache: 35-70MB model fits in L3 cache = 10-100x less energy
7. Battery-aware UI: show battery level, power saving mode option

Benchmark metrics: mWh/token, energy per task, battery % consumed, thermal curve
