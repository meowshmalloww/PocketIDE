--------------------------------------------------------------------------------
1. DEVELOPMENT ENVIRONMENT
--------------------------------------------------------------------------------

IDE: Android Studio (latest version)
  - Free download from developer.android.com/studio
  - Includes: Kotlin compiler, Gradle build system, Android SDK, emulator
  - Required for: NDK integration, CMake, JNI development
  - This is NOT optional — it's the standard Android development IDE

NDK: Android NDK r28+ (side-by-side install via SDK Manager)
  - Required for C++ native code (JNI bridge to ExecuTorch)
  - Required for hardware-level optimization (NEON, i8mm, SME2)
  - Set ANDROID_NDK environment variable

Build System: Gradle + CMake
  - Gradle: Kotlin/Java app build
  - CMake: C++ native library build (JNI bridge, ExecuTorch runtime)

--------------------------------------------------------------------------------
2. APP LANGUAGE & UI FRAMEWORK
--------------------------------------------------------------------------------

Language: Kotlin (primary) + C++ (native hardware layer)
  - Kotlin: all app logic, UI, orchestration, sandbox management
  - C++: JNI bridge to ExecuTorch, hardware instruction detection

UI Framework: Jetpack Compose + Material 3
  - Modern Android UI toolkit (replaces XML layouts)
  - Declarative UI (similar concept to Flutter widgets)
  - Built-in animations, state management
  - Dark theme support

Code Editor: tree-sitter + ComposeSyntaxHighlighter
  - tree-sitter: incremental parsing, syntax validation
  - ComposeSyntaxHighlighter: Compose Multiplatform syntax highlighting
  - Supports: Python, JavaScript, Java, and more
  - minSdk 26 (matches our target)
  - Editable text field with live highlighting

Layout: Split view (like Cursor/Windsurf)
  - Left panel: Chat interface (user prompt + AI response)
  - Right panel: Code editor (generated/editable code)
  - Bottom panel: Console output + error display
  - Top bar: Agent status indicators (colored), battery, settings

--------------------------------------------------------------------------------
3. INFERENCE FRAMEWORK
--------------------------------------------------------------------------------

PRIMARY: ExecuTorch [PENDING USER CONFIRMATION]
  - PyTorch's on-device solution, .pte format
  - KleidiAI integration via XNNPACK (Arm-optimized micro-kernels)
  - 50KB runtime footprint
  - Arm's official learning path uses ExecuTorch + KleidiAI + XNNPACK
  - Judge Michael Hall teaches this exact framework
  - Maven Central: org.pytorch:executorch-android
  - JNI bindings included in AAR
  - Supports INT4 groupwise quantization (8da4w mode)
  - mmap loading (LOAD_MODE_MMAP default)
  - LlmModule API: load(), generate(), token streaming via LlmCallback

  Key API:
    // Load model
    LlmModule module = new LlmModule(modelType, modelPath, tokenizerPath, temp);
    module.load();

    // Generate (async, token-by-token)
    module.generate(prompt, seqLength, callback);

  Build AAR from source:
    sh scripts/build_android_library.sh
    (requires ANDROID_HOME, ANDROID_NDK, ANDROID_ABI=arm64-v8a)

FALLBACK: LiteRT-LM (Google AI Edge)
  - If ExecuTorch export fails with Qwen models
  - Kotlin API: com.google.ai.edge.litertlm:litertlm-android
  - Supports Qwen2.5-0.5B confirmed (521MB, 30 decode tk/s on S24)
  - GPU + NPU acceleration
  - Multi-Token Prediction (MTP) for decode speedup
  - LoRA support for on-device inference

4. AI MODELS (THE MICRO-SWARM)
--------------------------------------------------------------------------------

BASE MODEL: Qwen2.5-Coder-0.5B-Instruct [PENDING USER CONFIRMATION]

  - Code-specific training, Llama-compatible architecture
  - HumanEval: 28% pass@1 | MBPP: 52.9% pass@1

LOW-END FALLBACK: SmolLM2-135M-Instruct

EXECUTORCH-TESTED FALLBACK: Llama 3.2 1B Instruct

  - Larger (~780MB INT4) but guaranteed to work

AGENT ROLES (3 specialized models):

  - Validator (orange): error trace → repair instructions

DEPLOYMENT:

--------------------------------------------------------------------------------
5. MODEL TRAINING / FINE-TUNING
--------------------------------------------------------------------------------

WHERE: Cloud GPU (A100, L40) or local (4080M)
  - All sufficient for 0.5B LoRA fine-tuning
  - A100/L40 = fastest, 4080M = good for experimentation

TOOLS:
  - Hugging Face TRL (SFTTrainer) + PEFT (LoRA)
  - Agent distillation: Qwen2.5-Coder-32B as teacher → 0.5B as student
  - Datasets: HumanEval, MBPP, CodeAlpaca, the-stack-smol

LoRA CONFIG (for 0.5B model):
  - Rank: r=16, alpha=32, dropout=0.05
  - Target: q_proj, v_proj (or all linear layers for better quality)
  - Optimizer: paged_adamw_8bit
  - Training: single GPU, hours not days

QUANTIZATION (POST-TRAINING):
  - INT4 groupwise (group_size=32) via ExecuTorch export
  - Compare: FP32 → INT8 → INT4 (show size/speed/quality tradeoffs)

EXPORT:
  python -m examples.models.llama.export_llama \
    --model "qwen2_5" --checkpoint <ckpt> --params <params> \
    --use_sdpa_with_kv_cache -X --xnnpack-extended-ops \
    --preq_mode 8da4w_output_8da8w --preq_group_size 32 \
    --max_seq_length 2048 -kv -d fp32 \
    --preq_embedding_quantize 8,0 \
    --metadata '{"get_bos_id":151643,"get_eos_ids":[151643]}' \
    --output_name "pocketide_coder_int4.pte"

7. ARM OPTIMIZATION STACK
--------------------------------------------------------------------------------

KleidiAI (Arm's AI acceleration):
  - Optimized micro-kernels for Arm CPU
  - Integrated into ExecuTorch via XNNPACK
  - Uses: NEON SIMD, i8mm, SME2 (when available)
  - Enable: -DEXECUTORCH_XNNPACK_ENABLE_KLEIDI=ON

Runtime Feature Detection:
  - NEON: always available (ARMv8-A+)
  - i8mm: detect via /proc/cpuinfo "i8mm" flag (ARMv8.6-A+)
  - SME2: detect via /proc/cpuinfo "sme2" flag (ARMv9.2-A+)
  - Graceful fallback: SME2 → i8mm → NEON

XNNPACK:
  - Default backend for ExecuTorch on Android
  - Quantized operations support
  - 4-thread CPU acceleration

Battery Efficiency:
  - Adaptive core selection: big cores for prefill, little for decode
  - Thermal monitoring: Android thermal service API
  - INT4 = 8x less memory access = proportional energy savings
  - Sequential execution = lower peak power
  - Battery tracking: Android BatteryManager API

Benchmarking:
  - Built-in benchmark mode in app
  - Arm LLM Runner (github.com/Arm-Examples/LLM-Runner) for standardized tests
  - Android Studio Profiler for CPU/memory analysis
  - Custom metrics: tokens/sec, TTFT, RSS, mWh/token, thermal curve
