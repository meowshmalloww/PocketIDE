1. BATTERY EFFICIENCY RESEARCH
================================================================================

The hackathon explicitly mentions "battery efficiency" as a Track 3 optimization
target. This is a scoring opportunity we must not miss.

--- KEY RESEARCH FINDINGS ---

EnerInfer (2026, arxiv.org/html/2606.23001v2):
- First on-device LLM framework that jointly manages energy, throughput, thermal
- Improves energy efficiency by up to 65% on phones
- Key insight: "modestly lowering NPU and DDR frequencies preserves quality of
  experience while substantially improving energy efficiency"
- Reduces overall device energy consumption by up to 11% vs OS defaults

FUSE (2025, arxiv.org/abs/2507.02135):
- Unified energy-aware governor for LLM inference on mobile
- Mobile DVFS governors (CPU, GPU, memory) operate independently → up to 40.4%
  longer latency vs optimal frequency combinations
- FUSE reduces TTFT by 7-17%, time-per-output-token by 25-37% with same energy

MNN-AECS (2025, arxiv.org/html/2506.19884):
- Adaptive Energy-Centric Core Selection for LLM decoding
- LLM decode phase is MEMORY-BOUND → doesn't need big CPU cores
- Dynamically selects low-power CPU cores during decode
- 23% energy reduction with no slowdown (averaged across 7 devices)
- 39-78% energy saving vs llama.cpp, ExecuTorch, MediaPipe, mllm
- NO ROOT ACCESS REQUIRED — works at application level
- Integrated into MNN framework (Alibaba's mobile inference engine)

llm.npu (2024, arxiv.org/html/2407.05858v2):
- NPU offloading for prefill phase
- 30.7x energy savings on average
- 22.4x faster prefill speed
- 1000+ tokens/sec prefilling for billion-sized models

MELT Study (2024, arxiv.org/html/2403.12844v4):
- First systematic study of on-device LLM execution
- Energy: 0.16-0.21 mWh per token
- ~500 prompts until battery depletion (40 input tokens, 135 output tokens)
- Galaxy S23: sustained power <8.5W, instantaneous max 14W
- iPhone 14 Pro: sustained 13.8W, instantaneous max >18W
- KEY: "continuous execution of LLMs remains elusive due to energy + thermal"
- Thermal throttling causes significant performance degradation over time

--- PRACTICAL BATTERY OPTIMIZATIONS FOR POCKETIDE ---

1. SEQUENTIAL EXECUTION (already planned):
   - Only one model loaded at a time = lower peak power draw
   - Between agents: CPU can downclock, saving energy
   - Natural energy management without extra code

2. INT4 QUANTIZATION (already planned):
   - Smaller model = less memory access = less energy
   - Memory access dominates LLM energy consumption
   - INT4 vs FP32: ~8x less data movement → proportional energy savings

3. ADAPTIVE CORE SELECTION (NEW — add to plan):
   - Use big cores (Cortex-X2/A76) for prefill (compute-bound)
   - Use little cores (Cortex-A55/A510) for decode (memory-bound)
   - Decode doesn't need big cores — it's waiting on memory
   - This is the MNN-AECS approach: 23% energy savings, no speed loss

4. THERMAL-AWARE INFERENCE (NEW — add to plan):
   - Monitor device temperature
   - Reduce thread count or clock speed when thermal throttling starts
   - Short bursts with cooldown periods > sustained heavy inference
   - 44% performance drop from thermal throttling within minutes

5. MMAP LOADING (already planned):
   - Memory-mapped file loading = only load needed weights
   - Reduces memory pressure = less energy on memory management
   - ExecuTorch supports LOAD_MODE_MMAP by default

6. SMALL MODELS IN CACHE (already planned):
   - 50M-135M model INT4 = ~35-70MB → fits in L3 cache
   - Cache access uses 10-100x less energy than DRAM access
   - This is a genuine optimization story for the judges

7. BATTERY-AWARE UI (NEW — add to plan):
   - Show battery level in the app
   - Warn user when battery is low
   - Option to switch to "power saving mode" (fewer agents, smaller context)
   - Track energy consumption per inference for benchmarking

IMPORTANT: Your LG Q70 actually uses Snapdragon 675, NOT 765.
(GSMArena confirms: LG Q70 = Qualcomm SDM675 Snapdragon 675, 11nm)

Specifications:
- CPU: 2x Cortex-A76 (2.0GHz) + 6x Cortex-A55 (1.7GHz)
- Architecture: ARMv8.2-A (64-bit)
- GPU: Adreno 612
- RAM: 4GB LPDDR4x
- OS: Android 9 (upgradable to Android 10), you said Android 12
- Process: 11nm

ARM Feature Support:
- NEON (SIMD): YES — all ARMv8-A CPUs have NEON
- i8mm (INT8 matrix multiply): NO — requires ARMv8.6-A+
- SME2 (Scalable Matrix Extension 2): NO — requires ARMv9.2-A
- SVE/SVE2: NO

CHALLENGE: 4GB RAM is EXTREMELY tight for LLM inference
- Android OS uses ~1.5-2GB
- Available to apps: ~1.5-2GB
- 0.5B model INT4: ~130MB weights + ~50-200MB KV cache + runtime overhead
- It MIGHT work but barely — need ultra-small model or aggressive optimization
- Using SmolLM2-135M INT4 (~35MB) would be safer on this device

OPPORTUNITY: If we get it working on this device, it's an INCREDIBLE WOW factor
- "Runs on a 2019 budget phone with 4GB RAM" = maximum optimization story
- Judges will be impressed if we show flat memory ceiling on 4GB device
- This is the ultimate proof of optimization
