1. INFERENCE FRAMEWORKS FOR ANDROID
================================================================================

--- ExecuTorch (PyTorch) ---
Source: executorch.ai, github.com/pytorch/executorch, learn.arm.com

- PyTorch's unified on-device AI solution
- 50KB runtime footprint, 12+ hardware backends
- .pte (PyTorch ExecuTorch) format — ahead-of-time compiled graphs
- Export pipeline: torch.export() → quantize/optimize → to_executorch() → .pte
- XNNPACK backend = default for Android CPU
- KleidiAI integration via XNNPACK (-DEXECUTORCH_XNNPACK_ENABLE_KLEIDI=ON)
- Supports Llama, Qwen (Llama-compatible architecture), and more
- JNI bindings for Android (build_android_library.sh script)
- Official Android demo app: examples/demo-apps/android/LlamaDemo
- Arm's official learning path uses ExecuTorch + KleidiAI + XNNPACK

Key export command for INT4 quantization:
  python -m examples.models.llama.export_llama
    --model "qwen2_5" --checkpoint <ckpt> --params <params>
    --use_sdpa_with_kv_cache -X --xnnpack-extended-ops
    --preq_mode 8da4w_output_8da8w --preq_group_size 32
    --max_seq_length 2048 -kv -d fp32
    --preq_embedding_quantize 8,0
    --metadata '{"get_bos_id":151643,"get_eos_ids":[151643]}'
    --output_name "model.pte"

Known issue: Qwen2.5-0.5B SpinQuant export had an assertion error with
qc4w conversion (github issue #9353). Workaround: use QAT or standard PTQ.
HuggingFace optimum-executorch supports Qwen2.5-0.5B with XNNPACK fp32 recipe.

--- LiteRT-LM (Google AI Edge) ---
Source: developers.google.com/edge/litert-lm, github.com/google-ai-edge/litert-lm

- Formerly TFLite, now production-ready LLM framework
- Cross-platform: Android, iOS, Web, Desktop, IoT
- GPU + NPU acceleration
- Multi-Token Prediction (MTP) for decode speedup (speculative decoding)
- Kotlin API: com.google.ai.edge.litertlm:litertlm-android
- Supports: Gemma, Llama, Phi-4, Qwen models
- MediaPipe LLM Inference API now in maintenance mode → migrate to LiteRT-LM
- LoRA support for on-device inference (Android supports static LoRA at init)
- Powers on-device AI in Chrome, Chromebook Plus, Pixel Watch

Benchmarks (Samsung S24 Ultra, CPU):
  Qwen2.5-0.5B:  251 prefill tk/s,  30 decode tk/s,  521MB model
  Gemma3-1B:     177 prefill tk/s,  33 decode tk/s, 1005MB model
  Qwen3-0.6B:    165 prefill tk/s,   9 decode tk/s (Vivo X300 Pro)
  FunctionGemma: 2238 prefill tk/s, 154 decode tk/s, 289MB (S25 Ultra)

Benchmarks (Samsung S26 Ultra, GPU):
  Gemma4-E2B: 3808 prefill tk/s, 52 decode tk/s, TTFT 0.3s

--- llama.cpp ---
Source: github.com/ggml-org/llama.cpp, learn.arm.com

- C/C++ LLM inference engine, CPU-focused
- KleidiAI integrated by default on Arm systems
- SME2 support: KleidiAI dispatches SME2-optimized kernels when available
- GGUF model format
- JNI bindings available (Arm-Examples/LLM-Runner)
- Benchmark (Llama 3.2 1B, 5 threads): 204 prefill tk/s, 48 decode tk/s

--- MLC LLM ---
Source: github.com/mlc-ai/mlc-llm, llm.mlc.ai

- Universal LLM deployment engine
- Android support via OpenCL GPU
- Qwen2.5-Coder-0.5B q4f16_0 on Snapdragon 8 Gen 1: 4.5 tok/s
  (q4f16_1 format was 0.7 tok/s — layout matters enormously on Adreno GPU)
- OpenAI-compatible API

2. ARM OPTIMIZATION TECHNOLOGIES
================================================================================

--- KleidiAI ---
Source: arm.com/markets/artificial-intelligence/software/kleidi, huggingface.co/Arm

- Arm's software-level AI acceleration (comprises KleidiAI + KleidiCV)
- Lightweight libraries with highly optimized micro-kernels
- Integrated into: ExecuTorch, llama.cpp, MediaPipe (via XNNPACK), ONNX Runtime
- Tailored to Arm CPU architectures
- 350+ tokens/sec on prefill for generative AI on mobile (Arm's claim)
- Uses NEON SIMD, i8mm, and SME2 instructions

--- SME2 (Scalable Matrix Extension 2) ---
Source: learn.arm.com/learning-paths/mobile-graphics-and-gaming/performance_llama_cpp_sme2

- Arm v9 architectural feature for matrix-heavy workloads
- Available on newer Arm v9 CPUs (Cortex-X3+, Cortex-A715+)
- Performance uplift measured on Vivo X300 Android phone:

  | Framework  | Model           | Threads | Without SME2 | With SME2 | Uplift  |
  |------------|-----------------|---------|--------------|-----------|---------|
  | mnn        | llama-3.2-1B    | 1       | 196 tk/s     | 339 tk/s  | 72.96%  |
  | mnn        | llama-3.2-1B    | 2       | 275 tk/s     | 396 tk/s  | 44.00%  |
  | mnn        | qwen25vl-3b     | 1       | 85 tk/s      | 134 tk/s  | 57.65%  |
  | llama.cpp  | llama-3.2-1B    | 1       | 148 tk/s     | 173 tk/s  | 16.89%  |
  | llama.cpp  | llama-3.2-1B    | 2       | 124 tk/s     | 191 tk/s  | 54.03%  |
  | llama.cpp  | phi-2           | 1       | 58 tk/s      | 77 tk/s   | 32.76%  |

Key insight: SME2 delivers significant boost even on single CPU thread.

--- i8mm (8-bit integer matrix multiply) ---
- Arm v8.2-a+ feature (most modern phones)
- Accelerates INT8 matrix multiply operations
- Required for ExecuTorch KleidiAI integration
- Arm learning path requires "Arm-powered smartphone with i8mm feature"

--- XNNPACK ---
- Google's floating-point neural network library, optimized for Arm
- Default backend for ExecuTorch on Android
- Supports quantized operations
- 4-thread CPU acceleration

--- NEON (ARM SIMD) ---
- Available on all modern Arm CPUs
- 128-bit SIMD vector operations
- Used by all KleidiAI kernels

--- Arm Performix ---
Source: developer.arm.com/servers-and-cloud-computing/arm-performix

- Arm's performance profiling and analysis toolkit
- Free to use
- Primarily designed for Neoverse/cloud workloads
- Profiles running applications, collects system + hardware-level metrics
- May have limited mobile support — use Android Studio Profiler as complement

--- Arm LLM Runner ---
Source: github.com/Arm-Examples/LLM-Runner

- Framework-agnostic benchmark tool for LLM inference
- JNI bindings for Android
- Measures: encode time, decode time, TTFT, total latency
- Supports: llama.cpp, MNN, ExecuTorch backends
- Standardized benchmarking pipeline

--- Qwen2.5-Coder-0.5B-Instruct (PRIMARY CANDIDATE) ---
Source: huggingface.co/Qwen/Qwen2.5-Coder-0.5B, arxiv.org/html/2409.12186v2

- 0.49B params (0.36B non-embedding), Apache 2.0
- 24 layers, 14 Q heads / 2 KV heads (GQA), 32K context
- Architecture: RoPE, SwiGLU, RMSNorm, tied word embeddings
- Llama-compatible architecture (can use ExecuTorch Llama exporter)

Benchmarks:
  HumanEval:     28.0% pass@1
  HumanEval+:    23.8% pass@1
  MBPP:          52.9% pass@1
  MBPP+:         47.1% pass@1
  BigCodeBench:  40.4% (3-shot)

  Via Ollama (RTX 4090): 322 tokens/sec, 27.4% pass@1 on MBPP (500 tasks)

Quantized sizes (GGUF format):
  Q2_K:  415 MB
  Q4_0:  429 MB
  Q4_K_M: 491 MB
  Q8_0:  676 MB

On Android (LiteRT, Samsung S24 Ultra, CPU, dynamic_int8):
  Prefill: 250.73 tk/s | Decode: 29.97 tk/s | TTFT: 2.31s | Memory: 1363MB
  Model size: 521MB (INT8)

On Android (MLC, q4f16_0, Snapdragon 8 Gen 1):
  ~276MB model, 4.5 tok/s generation (GPU OpenCL)

--- SmolLM2-135M-Instruct ---
Source: huggingface.co/HuggingFaceTB/SmolLM2-135M, github.com/huggingface/smollm

- 135M params, trained on 2T tokens
- Fine-tuned with SFT + DPO (UltraFeedback)
- Supports: text rewriting, summarization, function calling
- INT4 quantized versions available (AutoAWQ, AutoRound)
  - fbaldassarri/HuggingFaceTB_SmolLM2-135M-Instruct-auto_awq-int4-gs64-sym
  - Quantized for CPU inference specifically
- Fine-tuning scripts available (TRL + PEFT, LoRA r=16)
- W4A16 version: ~0.85GB for 1.7B model (135M would be much smaller)

--- TinyCodeLM-150M ---
Source: huggingface.co/upiter/TinyCodeLM-150M

- 150M params, fully open-source
- Pretrained on web text + Python code
- Instruction tuned with LintSeq algorithm (edit sequence data)
- HumanEval pass@1: 6.1 (pretrained) → 13.4 (instruction tuned)
- MBPP pass@1: 5.4 → 19.4 (instruction tuned)
- Trained on single H100 in ~2 days

--- nanoCoder ---
Source: github.com/jeffelin/nanocoder

- 25M params (from scratch), trains on CPU in hours
- Modern architecture: RoPE, GQA, SwiGLU, RMSNorm
- 3 presets: 25M (50m), 61.5M (100m), 175.6M (200m)
- 50m preset: ~1.5GB RAM, 3000 tok/s on M4 Pro, trains in ~20 min to 3.5 hrs
- Python code only
- Could be useful for ultra-lightweight Architect agent

4. MULTI-AGENT ARCHITECTURE RESEARCH
================================================================================

--- OpenAI Swarm ---
Source: github.com/openai/swarm

- Educational framework for multi-agent orchestration
- Runs almost entirely on client-side
- Agents are lightweight, stateless
- Explicit handoff functions between agents
- No state stored between calls (good for sequential execution)

--- Sequential Execution Pattern ---
Source: User's Gemini analysis + general edge AI research

The key insight for mobile: running multiple models simultaneously causes OOM.
Solution: sequential execution with active memory pruning.

  Phase 1: Load Agent A → Execute → Unload (purge weights from memory)
  Phase 2: Load Agent B → Execute → Unload
  Phase 3: Load Agent C → Execute → Unload

This ensures flat memory ceiling regardless of swarm complexity.
The orchestrator is a deterministic state machine (NOT AI-driven) for minimal overhead.

--- Agent Distillation ---
Source: arxiv.org/pdf/2505.17612, github.com/Nardien/agent-distillation

- Transfer agentic behavior from large LLMs to small models (0.5B-3B)
- Small agents can match next-tier larger models
- Uses: first-thought prefix prompting + self-consistent action generation
- Teacher: Qwen2.5-32B-Instruct → Student: Qwen2.5-0.5B/1.5B/3B
- Training data: teacher agent trajectories (available on Hugging Face)
- Built on smolagents + TRL SFT trainer

--- Knowledge Distillation for Code ---
Source: arxiv.org/abs/2510.17598 (Reasoning Distillation for Code)

- Distill reasoning capabilities of VLLM into smaller model
- Structure-aware loss optimization
- 9% improvement on MBPP pass@1 (34%) for distilled model
- Small model learns step-by-step reasoning from large model

--- MoEKD (Mixture of Experts Knowledge Distillation) ---
Source: doi.org/10.6084/m9.figshare.31144822.v3

- Multi-expert distillation for compressed code models
- 35.8% improvement in adversarial robustness
- 13% enhancement in predictive performance
- Ultra-compact models maintain competitive performance at half size

5. SELF-CORRECTING CODE GENERATION
================================================================================

--- CYCLE: Learning to Self-Refine Code ---
Source: dl.acm.org/doi/10.1145/3649825

- Automated self-refinement workflow imitating iterative programming
- Phase 1: Generate code from problem description
- Phase 2: Execute + test → aggregate feedback
- Phase 3: Feed back (description + faulty code + error) → refine
- Stop conditions: pass tests, max iterations, no improvement

--- Test-in-the-loop LLM Repair (TiLLR) ---
Source: doi.org/10.69987/jacs.2024.40206

- "Failing test → patch → regression test" loop
- 15% repair rate within 10 attempts on QuixBugs
- Tests as acceptance oracle, budget as operational knob
- Compact patches that don't overfit

--- Self-Repair Limitations ---
Source: arxiv.org/pdf/2306.09896

- Self-repair gains are often modest
- Weaker models benefit MORE from stronger model's feedback
- Key finding: substituting GPT-4's self-feedback with human feedback
  increased repaired programs by 1.58x
- Implication: Validator agent quality matters enormously
  → Consider using a stronger model for validation, or fine-tune validator
    specifically on error analysis

--- FTDI Framework (Self-Healing Multi-Agent) ---
Source: github.com/UMENZZE/FTDI-Framework

- Budget-aware self-healing for multi-agent code generation
- Planner → Coder → Tester pipeline
- Tiered repair: T0 regex fast-patch → T1 constrained edit → T2 deep regeneration
- Auditor: 5 diagnostic dimensions (deviation, consistency, uncertainty, boundary, off-by-one)
- Stops early when budget exhausted

8. ON-DEVICE MEMORY MANAGEMENT
================================================================================

--- LLM in a Flash ---
Source: Medium/@anuj.dutt9

- Windowing: reuse previously activated neurons, only load past few tokens
- Reduces I/O requests for loading model weights
- Key for low-memory devices

--- LLM as a System Service on Mobile ---
Source: arxiv.org/html/2403.11805v1

- Context memory model with Claim/Release primitives
- Manage memory as system service
- Orthogonal to inference speed optimizations

--- LiteRT-LM Memory Management ---
Source: developers.googleblog.com

- Memory-efficient dynamic loading
- mmap-based model loading (don't load entire model into RAM)
- Multi-Token Prediction for decode speedup

--- Practical Memory Tips (from Android LLM developer) ---
Source: xckevin.com/en/blog/android-local-llm-litert-mediapipe

- mmap loading, realistic maxTokens budget, device-tier fallback = 3 most effective levers
- KV cache: ~50MB at 128 tokens, ~200MB at 512 tokens (Gemma 2B)
- Set maxTokens to 512 instead of 1024 → cuts memory in half, reduces OOM
- LiteRT GPU Delegate 15-30% faster than ORT OpenCL EP on Snapdragon
- MediaPipe LLM Inference: low integration cost, zero customization (fixed Top-K)
- For production: use LiteRT manual management, not MediaPipe high-level API

--- Thermal Throttling ---
Source: buildmvpfast.com/blog/on-device-llm-mobile-llama-ios-android-2026

- Sustained inference causes thermal throttling: 44% performance drop within minutes
- Samsung Galaxy S24 Ultra accounts for 70%+ of local LLM downloads
- Heat is the single biggest constraint on mobile LLMs (not RAM or model size)
- Implication: benchmark with cooldown periods, show sustained vs burst performance

9. FINE-TUNING WORKFLOW
================================================================================

--- Unsloth + ExecuTorch ---
Source: unsloth.ai/docs/basics/inference-and-deployment/deploy-llms-phone

- Collaborative workflow: Unsloth trains → ExecuTorch deploys
- Quantization-aware training (QAT) → deploy directly to edge devices
- Streamlined: train LLM → deploy to Android/iOS phone

--- LoRA Fine-Tuning (Qwen2.5-0.5B) ---
Source: docs.databricks.com/aws/en/machine-learning/ai-runtime/examples/tutorials/sgc-finetune-qwen2-0.5b

- LoRA reduces trainable parameters by ~99%
- Merge adapters: combine LoRA weights with base model
- Single GPU sufficient for 0.5B model
- HuggingFace TRL SFTTrainer + PEFT LoRAConfig

--- SmolLM2 Fine-Tuning ---
Source: github.com/huggingface/smollm/blob/main/text/finetuning/train.py

- Example script uses TRL SFTTrainer + PEFT
- LoRA config: r=16, alpha=32, dropout=0.05, target=q_proj+v_proj
- BitsAndBytes for 4-bit quantized training (load_in_4bit, nf4)
- Dataset: bigcode/the-stack-smol (Python subset)
- Optimizer: paged_adamw_8bit

--- On-Device LoRA (LiteRT) ---
Source: developers.google.com/edge/mediapipe/solutions/genai/llm_inference

- Android supports static LoRA during initialization
- Load base LLM + LoRA weights at startup
- LoRA weights added to attention layers
- Models: Gemma-2 2B, Gemma 2B, Phi-2 (Qwen support likely coming)
